## 三、爆炸范围分析

### `build_graph` → `phases/context_graph.py:54`
- 风险: **high** | 置信度: medium
- 原因: 核心图构建逻辑从直接扫描源码并解析 JS/Vue 文件，改为依赖 SQLite GraphDB.update(project_root) 后再反序列化。该改动会影响所有依赖 ContextGraph 的节点、边、importers 数据来源；若 GraphDB 的解析行为、过滤目录、异常降级策略或数据模型与原实现不完全一致，可能导致影响链路、调用关系和导入关系结果发生系统性变化。
- 建议: 补充与旧实现等价性的回归测试，覆盖 JS/TS/JSX/TSX/Vue、跳过目录、解析失败 fallback、importer 关系等场景；确认 GraphDB.update 与原 build_graph 的文件过滤和解析语义一致。

### `_graph_from_db` → `phases/context_graph.py:68`
- 风险: **high** | 置信度: low *(需人工确认)*
- 原因: 新增的 DB 到 ContextGraph 转换逻辑直接依赖 db.all_nodes()、db.all_edges()、db.import_edges() 的 tuple 字段顺序，并将 import_edges 的 row[1] 作为被导入文件、row[0] 作为 importer 写入 graph._importers。当前上下文未提供 GraphDB schema，若字段顺序或边方向与假设不一致，会导致节点属性、边关系或反向导入索引错误，影响后续 blast radius 分析结果。
- 建议: 人工核对 phases.sqlite_graph.GraphDB 的 all_nodes/all_edges/import_edges 返回字段顺序和边方向；优先改为返回具名结构或在此处增加明确的 schema 注释与单元测试。

### `load_graph` → `phases/context_graph.py:258`
- 风险: **high** | 置信度: medium
- 原因: load_graph 现在优先加载派生出的 SQLite DB，只要 DB 文件存在就不会读取 legacy JSON。若 DB 过期、损坏、schema 不兼容或与 JSON 内容不一致，会静默绕过 JSON，返回错误或陈旧的 ContextGraph，从而影响后续所有依赖缓存图的分析结果。
- 建议: 增加 DB schema/version 校验、mtime 新旧判断和异常 fallback；当 DB 加载失败或落后于 JSON 时应回退读取 legacy JSON，并记录告警。

### `build_graph` → `phases/context_graph.py:59`
- 风险: **medium** | 置信度: high
- 原因: build_graph 现在会在项目根目录下创建 .luna/cache/context-graph.db，并引入持久化副作用。此前该函数主要在内存中构图。该变化可能影响只读项目、权限受限目录、CI 沙箱、并发执行以及缓存污染场景。
- 建议: 确认调用方允许 build_graph 写入项目目录；必要时支持传入 cache 路径、禁用缓存或在写入失败时降级到内存构图。

### `_db_path_from_json` → `phases/context_graph.py:79`
- 风险: **medium** | 置信度: medium
- 原因: 新增 JSON 路径到 DB 路径的派生规则为同目录同 stem 后缀 .db。该规则需要与 build_graph 写入的 .luna/cache/context-graph.db 以及 save_graph/load_graph 的历史路径约定保持一致；如果调用方传入的 JSON 路径不是 .luna/cache/context-graph.json，load_graph 可能找不到 build_graph 生成的 DB，或误读同目录其他 DB。
- 建议: 确认缓存路径命名约定是否统一；建议将 DB 路径生成逻辑集中复用，避免 build_graph 与 load_graph 使用隐式不同规则。
