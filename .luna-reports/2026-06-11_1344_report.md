## 三、爆炸范围分析

### `build_graph` → `phases/context_graph.py:75`
- 风险: **high** | 置信度: medium
- 原因: build_graph 引入 SQLite 增量缓存后，缓存命中的文件会跳过解析，仅依赖后续从数据库恢复 _importers。该函数是风险传播的核心数据源，如果缓存中的文件哈希、文件路径或边数据不一致，可能导致 importer 关系缺失，从而低估受影响文件范围。
- 建议: 为 build_graph 增加增量缓存一致性测试：首次全量构建、无变更二次构建、修改 importer、修改 imported、删除文件、移动文件，并断言 graph._importers 与 graph.edges 与全量解析结果一致。

### `build_graph` → `phases/context_graph.py:84`
- 风险: **high** | 置信度: low *(需人工确认)*
- 原因: 缓存命中判断使用 cached_hashes.get(str(src))，同时后续 db.get_all_files()/get_all_edges() 也假设数据库使用绝对路径字符串。如果 GraphDB 内部存储的是相对路径或规范化方式不同，会导致所有缓存失效，或更严重地导致边合并失败、删除集合计算错误，影响 blast radius 结果准确性。
- 建议: 人工确认 GraphDB 的路径存储契约；建议统一在 GraphDB API 边界明确使用绝对路径或相对路径，并增加路径格式断言测试。

### `build_graph` → `phases/context_graph.py:106`
- 风险: **high** | 置信度: low *(需人工确认)*
- 原因: db.sync(deleted, reparsed, new_importers, project_root) 在合并缓存边之前执行，且 new_importers 只包含本轮重新解析文件产生的关系。如果 GraphDB.sync 将 new_importers 视为完整图而不是增量更新，可能删除或覆盖未重新解析文件的缓存边，导致后续 find_usages/风险传播漏报。
- 建议: 人工审查 GraphDB.sync 语义；若 sync 需要完整图，应先合并缓存边再同步，或修改 API 明确传入 reparsed 范围内需要替换的边。

### `cli` → `luna.py:277`

- 建议: 确认 phases.sqlite_graph.GraphDB 在发布包中可用，并补充一次缓存命中、缓存不可用、首次构建三种场景的端到端测试。

### `build_graph` → `phases/context_graph.py:116`

- 原因: 从 db.get_all_edges() 合并缓存边时假设返回 imported_abs、importer_abs 且方向为 imported -> importers。若数据库边方向与该约定不一致，最终 graph._importers 和 graph.edges 方向会反转，导致风险传播错误。

- 建议: 全局搜索 graph.edges 的使用方，确认均接受 source=importer、target=imported 的方向约定，并补充方向说明测试。

## 四、代码质量问题

### `phases/context_graph.py:111` — 缓存命中时直接跳过解析，只创建了文件节点，没有从缓存恢复符号节点等完整图信息，导致 build_graph 返回的 ContextGraph 在缓存命中场景下与全量解析结果不一致。
- 类型: logic_gap | 风险: **medium** | 置信度: high
- 依据: graph.nodes[rel] = GraphNode(id=rel, node_type="file", file=rel, name=rel) ... if db is not None and cached_hashes.get(str(src)) == new_hash: ... continue
- 建议: 如果要跳过解析，需要从 SQLite 中恢复该文件对应的完整节点和边；或者明确将 ContextGraph 缩减为只包含文件依赖关系，并调整所有下游调用避免依赖符号节点。

### `phases/context_graph.py:132` — 代码计算了文件内容哈希 new_hash，但同步 SQLite 缓存时没有把新哈希传入 db.sync，可能导致缓存哈希无法更新或后续缓存命中判断不准确。
- 类型: logic_gap | 风险: **medium** | 置信度: medium
- 依据: new_hash = hashlib.sha256(content_bytes).hexdigest()；后续 db.sync(deleted=deleted_abs, reparsed=reparsed, new_importers=graph._importers, project_root=str(root)) 未传递 new_hash 或哈希映射
- 建议: 将本轮重新解析文件的哈希映射传给 GraphDB.sync，或确保 GraphDB.sync 内部会基于 reparsed 文件重新计算并持久化哈希。

### `phases/context_graph.py:105` — 读取文件内容失败时已经先向 graph.nodes 写入文件节点，然后直接 continue，返回的图中会残留未解析、无依赖边的文件节点。
- 类型: logic_gap | 风险: **low** | 置信度: high
- 依据: graph.nodes[rel] = GraphNode(...)；try: content_bytes = src.read_bytes() except OSError: continue
- 建议: 读取失败时不要提前写入节点，或在 except 中移除该节点并记录告警，避免产生不完整图数据。
