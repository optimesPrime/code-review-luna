## 三、爆炸范围分析

### `build_graph` → `phases/context_graph.py:70`
- 风险: **high** | 置信度: medium
- 原因: build_graph 从每次完整解析改为基于 SQLite 的增量缓存；缓存命中时会跳过解析源文件，仅创建 file 节点，随后只从 DB 合并 imports 边。若 SQLite 中未完整保存或未正确恢复符号节点、组件节点、导出信息等上下文，后续影响分析可能缺失未改动文件中的符号关系，导致漏报 blast radius。
- 建议: 确认 GraphDB 持久化的数据范围是否覆盖 ContextGraph 所需的全部语义信息；若目前只缓存 import 边，应避免跳过会产生符号/组件节点的解析，或扩展 SQLite 缓存以恢复完整 graph.nodes、graph.edges、_importers 等结构，并增加冷/热缓存一致性测试。

### `build_graph` → `phases/context_graph.py:110`
- 风险: **high** | 置信度: low *(需人工确认)*
- 原因: 新增调用 GraphDB.sync(deleted, reparsed, new_importers, project_root)。当前 diff 未展示 GraphDB 的实现，无法确认 sync 是否只删除 reparsed 文件的旧边并保留 skipped 文件缓存边；如果 sync 将 new_importers 视为全量图数据，可能会删除未重解析文件的关系，导致影响链路大面积缺失。
- 建议: 人工审查 phases/sqlite_graph.py 中 GraphDB.sync 的语义，确保 reparsed 是增量更新集合而非全量替换；补充测试：修改一个文件后，未修改文件的 import 边仍能从缓存恢复。

### `build_graph` → `phases/context_graph.py:95`
- 风险: **medium** | 置信度: medium
- 原因: 缓存命中判断使用 cached_hashes.get(str(src))，其中 src 为 resolve 后的绝对路径；若 GraphDB 内部保存的是相对路径或未规范化路径，会导致缓存永远 miss，退化为全量解析，影响性能但通常不影响正确性。
- 建议: 统一 GraphDB 中 file path 的存储规范，建议全部使用 resolve 后的绝对路径或项目相对路径，并为路径规范化增加单元测试。

### `build_graph` → `phases/context_graph.py:120`
- 风险: **medium** | 置信度: medium
- 原因: DB 写入或读取异常被整体吞掉，随后仅在 db_read_ok=false 且存在 skipped 文件时重解析 skipped 文件。异常不会暴露给调用方，可能掩盖缓存损坏、schema 不兼容、写入失败等问题，使性能或结果稳定性问题难以及时发现。
- 建议: 至少记录 warning/debug 日志，包含 DB 路径和异常原因；对于 schema 不兼容可考虑删除并重建缓存。

### `propagate_risk` → `phases/risk_propagation.py:65`
- 风险: **medium** | 置信度: medium
- 原因: 风险传播从直接读取 graph._importers 改为调用 graph.find_usages(current)。如果 find_usages 的实现与 _importers 语义完全一致，则是封装改进；但若 find_usages 依赖 graph.edges 或只返回符号级 usage，传播结果可能与原先基于 importers 的文件级传播不同。
- 建议: 确认 ContextGraph.find_usages(current) 返回的是导入 current 的文件集合，并与 _importers.get(current, set()) 保持兼容；增加针对多层 import 链路的传播回归测试。

### `cli` → `luna.py:274`
- 风险: **medium** | 置信度: high
- 原因: CLI 移除了 context-graph.json 的 load/save，改为每次调用 build_graph，并依赖 build_graph 内部 SQLite 增量缓存。旧 JSON 缓存将不再被读取，可能影响首次运行性能和缓存迁移，但功能上会回退到 build_graph 的全量解析或 SQLite 缓存。
- 建议: 更新相关文档和缓存清理逻辑；如需要兼容旧缓存，应提供迁移或明确废弃 context-graph.json。

### `_scan_source_files` → `phases/context_graph.py:53`
- 风险: **low** | 置信度: high
- 原因: 新增扫描函数将 source file 路径 resolve 为绝对路径，并使用 relative_to(root) 过滤跳过目录。root 已 resolve，常规项目内文件路径处理正确；主要影响是文件遍历和缓存 key 的输入发生变化。
- 建议: 保留针对软链接、越界路径和跳过目录的测试，确保不会扫描 node_modules、dist、build 等目录。

### `_parse_file` → `phases/context_graph.py:66`
- 风险: **low** | 置信度: high
- 原因: 新增解析封装函数将 Vue/JS 解析和 regex fallback 逻辑抽取出来，行为基本等价于原 build_graph 内联逻辑，风险较低。
