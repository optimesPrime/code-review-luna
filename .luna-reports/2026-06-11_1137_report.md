## 三、爆炸范围分析

### `build_graph` → `phases/context_graph.py:75`
- 风险: **high** | 置信度: high
- 原因: 增量缓存命中时直接 continue，当前内存中的 ContextGraph 只补了文件节点和后续从 SQLite 恢复的 import 边，但未恢复缓存文件中的 symbol/component/function 节点及其它分析产物。依赖完整 graph.nodes 的 changed_symbols、usage 查找或后续审查上下文可能在未改动文件上丢失信息，导致影响范围漏报。
- 建议: 不要只缓存 file_hash 和 import edges；需要将解析出的节点、边、符号索引等完整持久化并在缓存命中时恢复，或在当前版本继续全量解析仅用 SQLite 做辅助缓存。至少增加端到端测试覆盖：首次构图后第二次无改动构图，graph.nodes/edges/_importers 与首次结果等价。

### `build_graph` → `phases/context_graph.py:106`
- 风险: **high** | 置信度: high
- 原因: 删除文件同步 SQLite 时只删除 files 表记录和 edges WHERE target = path。按照注释约定 source=被导入文件、target=导入方；如果被删除文件曾作为被导入方，则 edges.source 仍会残留。后续加载 DB 会把已删除文件重新放入 graph._importers 和 graph.edges，造成陈旧依赖边和错误的 blast radius。
- 建议: 删除文件时同时清理 edges WHERE source = ? OR target = ?。增加测试：a.ts import b.ts，构图后删除 b.ts 再构图，确认不存在指向或来自 b.ts 的边。

### `build_graph` → `phases/context_graph.py:52`
- 风险: **medium** | 置信度: medium
- 原因: build_graph 内部直接创建并访问 GraphDB 及其私有成员 db._conn，且没有显式关闭连接。CLI 现在每次都会调用 build_graph，可能导致连接生命周期、并发执行、测试隔离和跨平台文件锁问题扩大。
- 建议: 为 GraphDB 增加公开 API 或上下文管理器，避免直接访问 _conn；在 build_graph 结束时关闭连接，或明确复用策略。补充并发/重复调用测试。

### `cli` → `luna.py:277`
- 风险: **medium** | 置信度: high
- 原因: CLI 前端图构建从 JSON cache 的 load/save 切换为每次调用 build_graph，并假设 build_graph 内部 SQLite 增量缓存完全可用。这改变了缓存格式、缓存失效路径和首/二次运行行为；如果 SQLite 缓存不完整或损坏，将直接影响后续 changed_symbols 与风险传播结果。
- 建议: 提供缓存迁移或回退策略，例如 SQLite 构建失败时回退全量构图；增加针对空缓存、损坏 DB、旧 JSON cache 存在时的 CLI 集成测试。

### `propagate_risk` → `phases/risk_propagation.py:65`
- 风险: **medium** | 置信度: medium
- 原因: 风险传播入口从直接读取 graph._importers.get(current) 改为 graph.find_usages(current)。如果 find_usages 不仅返回 importers，或其入参语义是符号/节点而非文件路径，传播结果会发生变化，可能扩大或遗漏影响链路。该改动与 context_graph 的 SQLite 边恢复同时发生，错误边会被进一步放大。
- 建议: 明确 find_usages(current) 的契约，并增加回归测试：文件 import 链、多层依赖、无使用场景下 propagate_risk 的输出应与预期一致；若只需要 importers，建议保留专用 API 或在 find_usages 中区分 edge_type。
